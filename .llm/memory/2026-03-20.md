## But du projet

Projet historique d'optimisation d'images via CloudFront + Lambda@Edge.

Flux principal :

1. `viewer-request-function` réécrit une URL source en URL d'alias cacheable.
2. `origin-response-function` intercepte les `404`/miss, récupère l'image originale depuis S3, applique les transformations Sharp, renvoie l'image transformée, et peut l'uploader dans S3.

Le travail actif doit désormais se concentrer sur les chemins hors `archives/*`.

## Architecture actuelle

### Viewer request

`lambda/viewer-request-function/index.js`

- Parse `alias`, `size`, `format`
- Normalise `jpg` vers `jpeg`
- Laisse passer l'original si aucun paramètre n'est demandé
- Laisse passer les GIF originaux sauf si on demande explicitement `webp`
- Réécrit l'URI en `..._<alias>_<size>.<format>`
- Réécrit le querystring avec `querystring.stringify(...)`

### Origin response

`lambda/origin-response-function/index.js`

- Si CloudFront/S3 renvoie `200`, la fonction retourne directement la réponse
- Si alias demandé :
  - déduit le `siteaccess` depuis l'URI
  - charge `conf-aliases-<siteaccess>`
  - lit `alias`, `size`, `format`, `original`
  - valide `size`
  - déduit le format original depuis l'extension
  - récupère l'image depuis S3
  - applique les filtres Sharp
  - encode la sortie en `jpeg`, `webp`, ou `avif`
  - retourne la réponse binaire base64

### Config des alias

`src/conf/common.js` contient :

- `Sizes`
- `Formats`
- quelques alias globaux

Les alias métier/site sont surtout dans `src/conf/aliases/`.

### Local env

`local-env/express-app/app.js` est une copie locale du comportement Lambda.
